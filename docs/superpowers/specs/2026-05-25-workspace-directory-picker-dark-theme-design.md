# Workspace Directory Picker Dark Theme Design

## Goal

将首页从“手动输入 workspace 路径”改为“通过目录选择面板选择目录”，并把首页及相关入口统一调整为偏黑色的控制台风格。

## Problem

当前首页要求用户直接输入本地 workspace 根目录路径，这有几个问题：

1. 交互粗糙，用户必须记住或粘贴绝对路径。
2. 新用户不容易理解应该输入什么，也不清楚路径是否有效。
3. 视觉风格与当前控制台式产品定位不够一致，首页仍有较强的浅色卡片感。

## Scope

本次设计只覆盖以下范围：

1. 首页 workspace 创建入口的交互改造。
2. 新增一个黑色目录选择面板。
3. 新增后端只读目录浏览接口，用于弹层逐级浏览目录。
4. 首页与该入口相关区域的整体深色化视觉调整。

不包含以下内容：

1. 整个产品所有页面的完整主题重构。
2. 原生系统文件选择器接入。
3. Electron、Tauri 或其他桌面壳能力。
4. workspace 创建后的 session 区域大改版。

## User Experience

### Homepage

首页主区域改为一个深色控制台风格的 hero 区，主动作从路径输入框切换为“选择目录”按钮。

用户流程：

1. 用户进入首页。
2. 点击“选择目录”按钮。
3. 打开黑色目录选择面板。
4. 从当前仓库目录开始浏览，也可以继续切换到整台机器的任意目录。
5. 选中一个目录后，面板底部显示绝对路径。
6. 点击“Use This Directory”关闭弹层。
7. 首页显示已选路径摘要，并提供“创建或打开 Workspace”动作。
8. 调用现有 `POST /api/workspaces` 完成创建或复用。

最近 workspace 列表仍保留，但从主入口降级为次级入口。

### Directory Picker Modal

目录选择面板是一个深色弹层，不使用整树递归，而是逐级浏览。

面板结构：

1. 顶部区域：
   - 标题
   - 关闭按钮
   - breadcrumb 路径条
   - “仓库根目录”快速跳转按钮

2. 中间区域：
   - 当前目录下的直属子目录列表
   - 仅显示目录，不显示文件
   - 每一行支持进入该目录

3. 底部区域：
   - `Selected Path`
   - `Cancel`
   - `Use This Directory`

交互规则：

1. 默认起始目录是当前仓库目录。
2. 用户可沿着父子层级浏览到任意系统目录。
3. 文件不可选，只有目录可确认。
4. 进入目录不会自动确认，必须显式点击 `Use This Directory`。
5. 用户关闭弹层后，不会丢失首页上一次已经确认的有效路径。

### Error States

面板内联展示错误，不通过全局 toast 曝光。

需要覆盖的错误：

1. 目录不存在。
2. 没有权限读取目录。
3. 后端返回非法路径错误。
4. 网络请求失败。

错误时：

1. 弹层保持打开。
2. 当前浏览状态尽量保留。
3. 如果此前已有一个有效的 `selectedPath`，不清空它。

## Visual Direction

整体视觉方向是偏黑色、偏控制台，而不是简单把白底改深灰。

### Palette

建议使用以下方向：

1. 背景：接近黑色的分层底色，例如 `#0b0d10`、`#11161c`、`#151a20`
2. 边框：低对比度冷灰蓝边框，例如 `#232b35`、`#28313c`
3. 正文：接近白色但不过曝，例如 `#f3f5f7`
4. 次级文本：中冷灰，例如 `#8d99a8`、`#96a0ad`
5. 高亮动作：浅色实心主按钮，形成强焦点

### Surface Treatment

1. 首页主卡片、目录弹层、最近 workspace 列表卡片统一用深色 layered surface。
2. 使用轻微渐变和细边框，不用单调纯黑平面。
3. 保持 Space Grotesk 与 IBM Plex Sans，延续现有字体系。

### Visual Hierarchy

1. “选择目录”是首页最强主动作。
2. 最近 workspace 列表保留，但弱于主 hero。
3. 已选路径使用等宽或接近等宽的表现强调“这是系统路径”。

## Backend Design

### New API

新增一个只读目录浏览接口，建议为：

`GET /api/workspaces/browse`

query 参数：

1. `path`：可选，当前要浏览的目录绝对路径

行为：

1. 当 `path` 缺失时，默认从当前仓库目录开始。
2. 当 `path` 存在时，校验该路径是否存在、是否为目录、是否可读取。
3. 成功时返回：
   - 当前目录绝对路径
   - 父目录绝对路径（如有）
   - breadcrumb 所需路径段
   - 当前目录下的直属子目录列表

返回中不包含文件内容，也不返回递归树，避免 payload 过大和权限暴露过多。

### Service Boundary

目录浏览逻辑放在 `workspaceService` 或与其同层的新只读 workspace browse service 中。

要求：

1. 使用 Node 文件系统 API 读取目录。
2. 仅返回 DTO，不把底层文件系统细节直接暴露到 route。
3. 对无权限、路径不存在、非目录情况统一转成 `ServiceError`。

## Frontend Design

### State Model

首页和目录面板状态拆分：

首页持有：

1. `selectedRootPath`
2. 目录面板开关状态
3. 创建 workspace mutation 状态

目录面板内部持有：

1. `currentPath`
2. `selectedPath`
3. `loading`
4. `error`
5. 当前浏览结果

这样弹层逻辑局部化，不把目录浏览细节塞进首页根组件。

### Data Flow

1. 首页点击“选择目录”。
2. 弹层打开，请求 `GET /api/workspaces/browse`，不带 `path`。
3. 后端返回当前仓库目录及其子目录。
4. 用户进入子目录时，请求 `GET /api/workspaces/browse?path=...`。
5. 用户确认目录后，将 `selectedPath` 回传给首页。
6. 首页调用现有 `createWorkspace({ rootPath })`。

### Component Structure

建议引入新的前端组件边界：

1. 首页 hero 区组件，负责主入口与路径摘要展示。
2. 目录选择弹层组件，负责浏览与确认。
3. 目录列表/路径 breadcrumb 子组件，按需要拆分，但不要过度工程化。

## Testing Strategy

### Backend

至少覆盖：

1. 默认起点返回当前仓库目录。
2. 指定有效目录时返回直属子目录列表。
3. 指定不存在路径时返回错误。
4. 指定文件路径而非目录时返回错误。
5. 权限失败时返回错误。

### Frontend

至少覆盖：

1. 点击首页按钮后打开目录选择面板。
2. 面板首次加载默认目录。
3. 点击子目录后进入下一层。
4. 点击确认后把绝对路径回传首页。
5. 创建 workspace 时使用已选路径。
6. 请求失败时在弹层中显示错误且不丢失已选路径。

## Risks

1. “浏览整台机器任意目录”意味着要谨慎处理权限失败和路径错误，否则体验会很差。
2. 如果前端一次性递归加载目录树，大目录会导致性能和响应问题，因此必须使用逐级加载。
3. 当前 `HomePage` 在 [apps/web/src/router.tsx](/home/daohaosisi/dev/OpenCode-Web-Lite-MVP/apps/web/src/router.tsx:37) 内较重，实现时应注意把新弹层逻辑局部化，避免继续堆大。

## Recommendation

采用方案 A：

1. 首页不再要求用户输入路径字符串。
2. 用黑色主按钮打开目录选择面板。
3. 用逐级浏览的服务端目录接口支撑目录选择。
4. 整体把首页和入口相关 surface 调整为更明确的深色控制台风格。
